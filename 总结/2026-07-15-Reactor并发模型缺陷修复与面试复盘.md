# TinyWebServer Reactor 并发模型缺陷修复与面试复盘

> 目标：用可核验的上游源码、提交历史和压测结果，说明我不是只把 TinyWebServer 跑起来，而是定位并修复了它原有 Reactor 路径中的并发设计缺陷。

## 1. 结论先行

**原始 TinyWebServer 确实具有同一个问题。**

这里的“同一个问题”准确指：**官方源码具有事件线程等待和无同步共享标志的数据竞争；在 GCC 11.4 的 `-O2` 构建下，该未定义行为会使 `improv` 等待判断被优化删除，并实际复现百次级低吞吐。**

> 2026-07-16 最终取证结论：历史构建命令包含 `-O2`。修复前提交 `a57a275 -O2` 实测 LT/LT 110、ET/ET 119 次/30 秒；官方 `4bcf887 -O2` 实测 149、440 次/30 秒。反汇编确认 `-O2` 删除了对 `improv` 的等待读取。最小 `atomic` 版本恢复到约 38 万至 41 万次但 CPU 仍为 114% 至 116%；正式异步修复保持约 38 万至 39 万次，并把 CPU 降到 71% 至 74%。

官方代码从 2020 年引入 Reactor 路径后，一直到本次检查的官方 `master`，读写事件中仍然保留以下结构：

```text
主事件线程：
    投递任务到线程池
    -> while (true) 原地轮询 improv
    -> 等工作线程修改共享标志位后才继续处理 epoll 中的其他事件
```

这会导致所谓的 Reactor 主事件循环在每次投递任务后停止分发事件，并占用 CPU 忙等工作线程。它破坏了事件循环应当具备的非阻塞分发能力，同时还包含未同步共享变量造成的 C++ 数据竞争。

我对这个问题的准确表述应该是：

> 我在复现和源码审计中定位到 TinyWebServer 官方上游仍保留的 Reactor 同步缺陷，并完成了异步化和生命周期安全改造。

不要说“这是我在全网第一个发现的 bug”。本次能够证明的是：**官方指定提交中存在该实现，我独立复现、定位、修复并验证了它**，而不是能够证明此前从未有人讨论过它。

## 2. 上游源码证据

### 2.1 主事件线程投递后原地自旋

官方 [`webserver.cpp` 第 279-355 行](https://github.com/qinguoyi/TinyWebServer/blob/4bcf88762f85135e9bd46a1032e815e458de6f2e/webserver.cpp#L279-L355) 中，Reactor 的读、写路径都具有同样结构：

```cpp
m_pool->append(users + sockfd, 0);

while (true)
{
    if (1 == users[sockfd].improv)
    {
        if (1 == users[sockfd].timer_flag)
        {
            deal_timer(timer, sockfd);
            users[sockfd].timer_flag = 0;
        }
        users[sockfd].improv = 0;
        break;
    }
}
```

写事件把状态参数换成 `1`，之后仍然执行相同的 `while (true)`。

问题不只是“循环看起来不优雅”，而是此时运行这段循环的线程正是唯一负责 `epoll_wait` 和事件分发的主线程。它等待一个连接时，其他已经就绪的连接也不能被继续分发。

### 2.2 工作线程通过普通整数发布状态

官方 [`threadpool/threadpool.h` 第 110-144 行](https://github.com/qinguoyi/TinyWebServer/blob/4bcf88762f85135e9bd46a1032e815e458de6f2e/threadpool/threadpool.h#L110-L144) 中，工作线程直接写入：

```cpp
request->improv = 1;
request->timer_flag = 1;
```

而官方 [`http/http_conn.h` 第 80-115 行](https://github.com/qinguoyi/TinyWebServer/blob/4bcf88762f85135e9bd46a1032e815e458de6f2e/http/http_conn.h#L80-L115) 把 `improv`、`timer_flag` 和 `m_state` 定义为普通 `int`，没有互斥锁、原子变量或其他完成通知机制。

`improv` 和 `timer_flag` 由工作线程写、主线程同时读，构成 C++ 内存模型下的数据竞争，程序行为未定义。即使在某次编译和某台机器上“看起来能跑”，也不能据此认为同步正确。

### 2.3 完成标志发布时机还早于真正处理完成

官方 Reactor 读成功分支的顺序是：

```cpp
if (request->read_once())
{
    request->improv = 1;
    connectionRAII mysqlcon(&request->mysql, m_connPool);
    request->process();
}
```

也就是说，名义上的完成标志在 `process()` 之前就被设置。主线程可能已经退出等待，但工作线程仍在解析 HTTP、访问数据库并生成响应。

这说明 `improv` 并不是一个语义可靠的“任务完成”事件，只是一个时序脆弱的跨线程标志。

### 2.4 提交历史证明不是本项目后加的回归

对官方完整 Git 历史执行字符串追踪：

```bash
git log -S "m_pool->append(users + sockfd, 0)" -- webserver.cpp
git log -S "request->improv = 1" -- threadpool/threadpool.h
git blame -L 286,355 webserver.cpp
git blame -L 116,142 threadpool/threadpool.h
```

两条关键路径都指向官方提交 `7a16b8f9`。本项目修复前提交 `a57a275` 继承了相同的自旋和共享标志结构，因此该问题来源于上游实现，不是本项目后来添加注释、数据库配置或压力测试时引入的。

## 3. 根因链，而不是只看“CPU 高”

### 3.1 事件循环被同步等待串行化

正确的 Reactor 目标是：

```text
epoll 返回一批就绪事件
    -> 主线程快速分发任务
    -> 立即继续处理下一事件或再次 epoll_wait
    -> 工作线程异步完成 IO/业务
```

当编译器实际保留轮询时，上游行为是：

```text
epoll 返回一批就绪事件
    -> 投递第一个任务
    -> 主线程自旋等第一个工作线程
    -> 等到共享标志改变后才分发下一个事件
```

因此，线程池虽然有多个工作线程，事件分发入口却被逐事件同步等待限制。并发结构退化成“投递一个、等一下、再投递下一个”。但这只描述 `-O0` 或使用原子变量后的可执行路径；历史 `-O2` 构建触发未定义行为后，编译器直接删除了等待条件，形成另一种更严重的活性故障，见第 6 节。

### 3.2 忙等同时与真正干活的线程争抢 CPU

当等待循环被保留时，`while (true)` 没有阻塞、让出或退避。尤其在单核或小规格云服务器上，主线程会持续占用时间片，工作线程反而更难及时获得 CPU 去修改 `improv`。

这形成了一个反直觉链路：

```text
主线程越着急等结果
    -> 自旋消耗越多 CPU
    -> 工作线程越晚运行
    -> 主线程等待越久
    -> 整体吞吐越低
```

历史 `-O2` 故障走的是另一条链：等待条件被删除，短暂错误执行后主线程和工作线程都进入内核等待，CPU 降到接近 0。因此“Reactor 模式 CPU 看起来不高”不能证明它高效，必须结合吞吐、完成请求数、线程等待点和反汇编一起判断。

### 3.3 普通变量跨线程读写是未定义行为

这不是简单的“可能读到旧值”。在 C++ 中，两个线程无同步地读写同一普通变量且至少一方写入，就是数据竞争，程序行为未定义。编译器没有义务保证主线程每轮都重新从内存读取 `improv`。

本次反汇编进一步确认，GCC 11.4 在 `-O2` 下没有生成对 `improv` 的循环读取，而是直接执行循环体中的状态检查和清零后返回。因此这里不是泛泛的“可能读到旧值”，而是优化后的控制流已经与源码直觉不同。

`volatile` 也不能解决这个问题，因为它不提供：

- 原子性；
- 线程间 happens-before 关系；
- 内存可见性顺序保证；
- 复合状态的一致性。

### 3.4 任务状态不应临时写进共享连接对象

上游 `append(request, state)` 会把读写状态写入 `request->m_state`，队列里只保存 `request` 指针。这让“本次队列任务的上下文”和“可复用连接对象的长期状态”耦合在一起。

更稳妥的设计是让每个队列项独立保存：

```cpp
struct work_item
{
    http_conn *request;
    int state;
    std::uint64_t generation;
};
```

这样，读写状态属于任务，不再依赖随后可能变化的连接成员。

## 4. 为什么不能只做一个表面修补

### 4.1 只把 `int` 改成 `std::atomic<int>` 不够

原子变量可以消除相应标志位的数据竞争，却不会改变主事件线程仍在原地等待的事实。事件循环依旧不能继续分发其他连接，因此架构瓶颈仍然存在。

### 4.2 用条件变量替换自旋也不够

条件变量会降低忙等 CPU，但主事件线程仍然会阻塞等待某一个任务。它解决的是“怎么等”，没有解决“事件循环不应该等”。

### 4.3 直接让工作线程关闭连接也不稳妥

连接关闭同时涉及：

- 从 epoll 删除 fd；
- 关闭 socket；
- 更新连接计数；
- 删除主线程维护的定时器；
- 防止 fd 被系统快速复用后，旧任务误操作新连接。

如果工作线程直接操作定时器链表和连接生命周期，就会把原本由主事件线程管理的资源变成多线程共享资源，进一步引入锁、悬空指针、重复关闭和 fd ABA 风险。

正确方向不是把自旋循环简单删除，而是重新明确成功路径、失败路径和资源所有权。

## 5. 我的修复方案

### 5.1 成功路径完全异步，不再回传“完成标志”

Reactor 主线程现在只做：

```text
收到 EPOLLIN/EPOLLOUT
    -> 刷新定时器
    -> 将 work_item 投递到线程池
    -> 立即返回事件循环
```

工作线程负责执行 Reactor 的 socket IO 和业务处理。成功时，现有 `read_once()`、`process()`、`write()` 路径根据下一阶段通过 `EPOLLONESHOT` 重新注册读或写事件。

这里不需要再额外通知主线程“成功了”，因为重新武装 epoll 本身就是后续状态推进机制。

### 5.2 失败路径使用 `eventfd` 回到主事件循环

工作线程处理失败时：

1. 把 `{sockfd, generation}` 放入互斥锁保护的失败队列；
2. 写 `eventfd`；
3. `eventfd` 已注册到同一个 epoll；
4. 主事件线程收到该事件后，统一关闭连接并删除定时器。

数据通道和唤醒通道被明确分开：

```text
失败详情：mutex + queue
唤醒通知：eventfd
资源回收：主事件线程
```

选择 `eventfd` 的原因是它是 Linux 为事件通知提供的轻量级文件描述符，可以直接接入 epoll；相比让主线程阻塞在条件变量上，它不会破坏统一事件循环。

### 5.3 队列项保存不可变任务上下文

线程池队列由 `std::list<T *>` 改为保存 `work_item`：

```cpp
struct work_item
{
    T *request;
    int state;
    std::uint64_t generation;
};
```

这样每个任务拿到的是投递时确定的读写状态和连接代次，不再通过 `http_conn::m_state` 临时传递任务参数。

### 5.4 用连接代次过滤迟到的失败通知

文件描述符只是一个可复用整数。例如旧连接 fd=10 关闭后，新连接可能很快再次得到 fd=10。

如果旧工作任务稍后报告“10 号连接失败”，只按 fd 处理就可能关闭新的连接。修复后，每次接收连接都会递增 `generation`，失败通知同时携带该值；主线程只处理 fd 和 generation 都匹配的通知。

准确地说，这一机制保护的是**迟到失败通知的关闭动作**，避免旧失败误关新连接；它不是对所有可能的旧任务访问做万能保护。

### 5.5 补齐线程和定时器生命周期

为了让异步化之后的资源生命周期可控，修复还包括：

- 线程从 detached 改为 joinable；
- 析构时设置停止状态、唤醒全部线程并 `pthread_join`；
- 先等待工作线程退出，再销毁连接数组等共享资源；
- 删除定时器后把对应指针置空；
- 异常事件处理前检查定时器是否仍存在，避免重复关闭；
- 任务队列满时立即走连接清理路径；
- 监听 backlog 从固定 `5` 调整为 `SOMAXCONN`，减少高并发连接建立阶段的排队瓶颈。

其中 backlog 调整是高并发配套优化，不是 Reactor 自旋缺陷本身。面试时必须把两者区分开。

## 6. 修复效果与证据边界

### 6.1 恢复历史构建条件

第一次复测使用官方 Makefile 的 `-g` 和本项目未指定优化级别的构建，实际相当于 `-O0`，所以没有复现旧结果。继续追查原始会话后，恢复出的历史命令明确包含：

```bash
g++ -std=c++11 -O2 -Wall \
  main.cpp timer/lst_timer.cpp http/http_conn.cpp log/log.cpp \
  CGImysql/sql_connection_pool.cpp webserver.cpp config.cpp \
  -o server -pthread $DB_FLAGS
```

修复前 `-O2` 重建 ELF 为 103744 字节，`ls -lh` 显示 102K，与历史截图大小一致。原 `server` 被 `.gitignore` 排除，现已无法进行 Build ID 或逐字节一致性证明，因此不能声称找回了当时完全相同的 ELF；但构建命令、源码结构、二进制大小和故障数量级均重新对应。

### 6.2 优化级别与最小同步消融

环境为同一台 2 核云服务器、100 个 Webbench 客户端、30 秒、默认 8 个工作线程、关闭日志。各组顺序运行，避免两套服务器互相争抢 CPU。

| 版本 | 模式 | 完成请求/30s | 吞吐 | CPU | 失败 |
|---|---|---:|---:|---:|---:|
| `a57a275 -O0` | LT/LT | 382123 | 12737.43 req/s | 115.77% | 0 |
| `a57a275 -O0` | ET/ET | 360313 | 12010.43 req/s | 117.11% | 0 |
| `a57a275 -O2` | LT/LT | **110** | **3.67 req/s** | **0.06%** | 0 |
| `a57a275 -O2` | ET/ET | **119** | **3.97 req/s** | **0.06%** | 0 |
| `a57a275 -O2 + atomic` | LT/LT | 408165 | 13605.50 req/s | 114.33% | 0 |
| `a57a275 -O2 + atomic` | ET/ET | 384738 | 12824.60 req/s | 115.99% | 0 |
| 正式修复版 `-O2` | LT/LT | 394167 | 13138.90 req/s | 74.42% | 0 |
| 正式修复版 `-O2` | ET/ET | 382802 | 12760.07 req/s | 71.33% | 0 |

最小 `atomic` 版本只把 `improv/timer_flag` 改成 `std::atomic<int>`，保留原等待循环。它证明正确同步能恢复活性和吞吐，但主事件线程仍在等待，CPU 仍高。

正式修复版相对最小 `atomic` 版本：

- LT/LT：CPU 降低 34.91%，吞吐变化 -3.43%；
- ET/ET：CPU 降低 38.50%，吞吐变化 -0.50%；
- 两组均 0 失败，压测后仍返回 HTTP 200。

因此，`atomic` 解决“数据竞争导致 O2 活性崩溃”，异步重构继续解决“事件线程等待和高 CPU”。

### 6.3 反汇编和线程状态确认机制

`improv` 位于 `http_conn` 对象偏移 `4`。`-O0` 和 `-O2 + atomic` 都生成反复读取该地址的循环：

```text
mov  0x4(...), %eax
cmp  $0x1, %eax
jne  回到读取指令
```

普通 `int` 的 `-O2` 版本在 `append()` 后不再读取偏移 `4`，而是直接检查偏移 `0` 的 `timer_flag`、清零 `improv` 并返回。这说明问题不是笼统的“缓存没刷新”，而是编译器已经基于未定义行为删除了等待条件。

故障发生后，进程采样和 GDB 调用栈显示：

```text
主线程：epoll_wait -> WebServer::eventLoop
8 个工作线程：sem_wait -> threadpool::run
```

这解释了低吞吐时 CPU 为什么接近 0：服务器不是高效，而是在初始少量请求后失去可推进的事件与任务。

### 6.4 官方固定提交完整实测

官方提交 `4bcf88762f85135e9bd46a1032e815e458de6f2e` 仅替换数据库连接配置，按同样 `-O2` 命令构建，实际运行结果为：

| 官方模式 | 完成请求/30s | 吞吐 | CPU | 失败 |
|---|---:|---:|---:|---:|
| Reactor LT/LT | **149** | **4.97 req/s** | **0.09%** | 0 |
| Reactor ET/ET | **440** | **14.67 req/s** | **0.16%** | 0 |

官方二进制反汇编也出现同样的 `improv` 等待删除，线程状态同样是主线程 `epoll_wait`、工作线程 `sem_wait`。这不是只看到相同代码后的推断，而是官方 Git blob 的完整构建和两轮 30 秒压测。数据竞争本身具有调度不确定性，所以不应要求每轮精确复现同一个百次数字；稳定证据是万级到百次级的数量级下降、近零 CPU、线程等待点和相同机器码结构。

### 6.5 恢复倍数如何正确表达

正式修复版相对本轮故障 `-O2` 版，完成请求数恢复为：

- LT/LT：`394167 / 110 = 3583.34` 倍；
- ET/ET：`382802 / 119 = 3216.82` 倍。

这可以作为**同机、同页面、同并发、同优化级别下消除活性故障的恢复倍数**。不能写成“通用 QPS 永久提升 3000 多倍”，因为正常工作的 `atomic` 版本和正式修复版吞吐处在同一数量级，正式方案更稳定的性能价值是 CPU 降低约 35% 至 39%。

### 6.6 实验边界

测试仍有以下限制：压测端与服务端位于同一台 2 核主机；每种模式只保留一轮正式结果；Webbench 不提供 P95/P99；正式修复还包含失败回收、generation、线程退出和 backlog 改动，不是单一补丁。

完整原始数据、ELF 信息、反汇编、线程状态、GDB 调用栈和脚本保存在：


## 7. 这个改动真正有价值的地方

### 7.1 从“会使用 epoll”进阶到“理解事件循环约束”

很多项目描述只会写“使用 epoll + 线程池实现高并发”。这次改动能够证明我理解：

- 事件线程不能等待单个任务，否则批量就绪事件会被串行化；
- 线程池存在不等于系统真的并发；
- Reactor/Proactor 的关键是 IO 和业务责任如何分配，而不是命令行参数叫什么；
- `EPOLLONESHOT` 需要与重新武装和单连接并发访问约束配合。

### 7.2 从“代码能跑”进阶到“遵守 C++ 内存模型”

修复不是把 `improv` 改成 `volatile`，而是取消错误的共享标志同步，让跨线程通知通过明确的同步原语完成。这能够体现对以下问题的理解：

- data race 与 race condition 的区别；
- happens-before；
- 原子变量能解决什么、不能解决什么；
- 为什么事件通知和共享数据仍需各自的同步策略。

### 7.3 从“修性能”进阶到“处理资源所有权”

删除自旋很容易，安全删除自旋并不容易。真正的工程工作是处理：

- 谁负责关闭 socket；
- 谁负责删除定时器；
- 失败怎么跨线程回到事件循环；
- fd 复用后如何识别迟到通知；
- 服务器退出时如何保证线程不再访问已释放资源。

这部分比单纯背诵 Reactor 定义更有面试价值。

## 8. 30 秒面试话术

> 我的项目基于 TinyWebServer。我恢复历史 `-O2` 构建条件后，在修复前提交和官方固定提交上都复现了 Reactor 百次级低吞吐。反汇编确认普通 `int improv` 的跨线程数据竞争使 GCC 删除了等待判断；最小 atomic 对照恢复吞吐，但 CPU 仍超过 114%。我没有停在改 atomic，而是移除主线程等待，用不可变任务、EPOLLONESHOT、eventfd 和 generation 完成异步化。正式修复在本轮恢复到 38 万至 39 万次/30 秒，并相对 atomic 版本把 CPU 降低约 35% 至 39%，两组均 0 失败。

## 9. 两分钟 STAR 话术

### S：背景与现象

> 我最初在云服务器上得到一组异常结果：Proactor 每秒上万请求，而 Reactor 只有个位数到十余 req/s。第一次用 `-O0` 重编译没有复现，但我没有停在“历史异常”，而是继续从原始会话恢复构建条件，发现当时命令实际包含 `-O2`。

### T：我的任务

> 我的目标不是把参数调高，而是确认结果是否真实、问题来自我的改动还是官方上游，并在不破坏定时器、EPOLLONESHOT 和 Proactor 的前提下修复它。

### A：定位与方案

> 我从 `eventLoop -> dealwithread/dealwithwrite -> threadpool::run` 追踪执行链，并用 Git 历史确认缺陷来自官方上游。随后建立 `-O0/-O2/volatile/atomic/正式修复` 消融矩阵，保存 ELF、反汇编、pidstat 和 GDB 全线程栈。机器码显示普通 `int` 的 `-O2` 版本完全没有读取 `improv` 的等待分支；atomic 版本则恢复了循环。正式方案把任务状态和 generation 放入不可变队列项，成功路径依靠 EPOLLONESHOT 推进，失败路径用 eventfd 回到主循环统一回收，并把线程改为 joinable 以保证退出顺序。

### R：结果与价值

> 修复前 `-O2` 的 LT/LT、ET/ET 分别只有 110 和 119 次/30 秒；官方 `-O2` 也只有 149 和 440 次。最小 atomic 版本恢复到 408165 和 384738 次，但 CPU 为 114% 至 116%；正式修复达到 394167 和 382802 次，CPU 降到 74.42% 和 71.33%，失败均为 0。故障恢复倍数约为 3583 和 3217，但我把它限定为同机同条件下消除活性故障的结果，不宣传为通用加速比。

## 11. 高频追问与回答

### Q1：为什么自旋会让 Reactor 退化？

因为自旋发生在唯一的 epoll 分发线程中。它等待当前任务时，不能处理同一批次后续事件，也不能及时再次进入 `epoll_wait`。多工作线程无法弥补分发入口被同步阻塞的问题。注意本次故障 `-O2` 二进制并没有真的持续自旋：数据竞争使等待条件被优化删除，服务器短暂错误执行后进入近零 CPU 的停滞状态。这是同一错误设计在不同构建下的两种表现。

### Q2：为什么不用 `std::atomic<bool>`？

atomic 只能让标志位读写符合内存模型，不能让事件循环继续工作。主线程仍在等待单个任务，架构上的串行化没有消失。

### Q3：为什么不用条件变量？

条件变量能避免耗 CPU，但仍会阻塞事件线程。Reactor 需要的是异步完成和事件驱动通知，而不是换一种阻塞等待方式。

### Q4：为什么使用 `eventfd`？

它是 Linux 提供的轻量级事件计数描述符，可以注册到 epoll，使工作线程通知、信号管道、监听 socket 和客户端 socket 都由同一个事件循环处理。失败详情仍放在互斥锁保护的队列中，`eventfd` 只负责唤醒。

### Q5：为什么成功路径不也写一次 `eventfd`？

成功处理后，HTTP 状态机会通过 `modfd` 重新武装 `EPOLLONESHOT`，下一次读写就绪自然会回到 epoll。再发一条成功通知只会增加上下文切换和事件处理。失败时需要主线程接管连接及定时器回收，才需要显式通知。

### Q6：`EPOLLONESHOT` 在这里有什么作用？

它保证某个连接事件被取出后，在显式 `modfd` 重新武装之前不会再次被 epoll 报告，从而避免多个线程同时处理同一个连接。工作线程完成当前阶段后，根据状态机把它重新注册为下一次 `EPOLLIN` 或 `EPOLLOUT`。

### Q7：generation 解决什么问题？

它解决失败通知处理阶段的 fd ABA：旧连接关闭后 fd 数字可能被新连接复用，只有 fd 和 generation 都匹配，主线程才执行关闭。它不是智能指针，也不能自动解决所有对象生命周期问题。

### Q8：为什么线程要从 detached 改成 joinable？

detached 线程无法在服务器析构时被可靠等待。如果先释放 `users`、连接池或回调所属对象，后台线程仍可能访问这些资源。joinable 线程允许先发出停止信号、唤醒线程、等待退出，再销毁共享资源。

### Q9：你怎么证明问题来自上游？

一是本项目 README 指向 `qinguoyi/TinyWebServer`；二是官方固定提交仍保留相同的读写等待和普通标志；三是 `git log -S` 与 `git blame` 将关键代码追溯到官方 2020 年的 Reactor 引入提交；四是我的修复前提交保留相同结构；五是我实际构建并压测官方 `4bcf887 -O2`，复现 149/440 次，并在反汇编中看到相同的 `improv` 等待删除。

### Q10：最初的千倍差距最后确认了吗？

确认了故障机制，但要限制表述。第一次 `-O0` 复测漏掉了历史命令中的 `-O2`；恢复该条件后，修复前提交复现为 110/119 次，正式修复为 394167/382802 次，对应 3583/3217 倍的活性恢复。这个数字只适用于本轮同机同条件故障对照，不能说成所有负载下的通用 QPS 提升。正常工作的 atomic 版本与正式修复版吞吐相近，正式重构的额外性能收益主要是 CPU 降低约 35% 至 39%。

### Q11：官方 README 也有很高的 QPS，为什么你的官方原版这么低？

关键变量是优化级别。官方 Makefile 的 `-g` 构建相当于 `-O0`，本机确实能达到约 1.2 万 req/s；按历史命令使用 `-O2` 后，普通共享标志的数据竞争触发未定义行为，反汇编显示 `improv` 等待被删除，官方 LT/LT、ET/ET 只剩 149 和 440 次/30 秒。这不是 Reactor 模型或 GCC 的错误，而是源码违反 C++ 内存模型后被优化暴露。

### Q12：这个项目现在就是生产级了吗？

不是。这次修复解决的是 Reactor 分发和相关生命周期问题。HTTP 解析边界、安全校验、路径访问、超时策略、可观测性、配置管理和更系统的并发测试仍需要继续完善。能主动说明边界比宣称“生产可用”更可信。

## 12. 现场画图版本

修复前源码意图，以及 `-O0/atomic` 下的实际路径：

```text
                    +----------------------+
epoll -> main ----> | append(request)      |
                    | while (!improv) {}   | <--- 主事件循环停在这里
                    +----------+-----------+
                               ^
                               | 普通 int，无同步
                               |
                    +----------+-----------+
                    | worker: read/process|
                    | improv = 1           |
                    +----------------------+
```

历史普通 `int + -O2` 的实际路径：

```text
epoll -> main -> append(request)
              -> 不读取 improv
              -> 过早检查 timer_flag
              -> 清零 improv 后返回
              -> 完成与失败时序失配
              -> main: epoll_wait / workers: sem_wait
```

修复后：

```text
epoll -> main -> append(work_item) -> 立即继续分发
                |
                v
            worker 执行 IO/业务
                |
                +-- 成功 -> modfd 重新武装 EPOLLONESHOT
                |
                +-- 失败 -> queue{fd,generation} + eventfd
                                                |
                                                v
                                  epoll 唤醒 main 统一回收
```

## 13. 后续如果继续增强，应做什么

1. 对每种模式重复至少 5 次，报告均值、标准差和置信区间。
2. 把自旋移除、`eventfd`、generation、线程退出和 backlog 改动拆成独立提交或实验开关，做消融测试。
3. 使用 `perf stat` 和 `pidstat -t` 记录每线程 CPU、上下文切换和调度行为。
4. 引入支持延迟统计的压测工具，记录平均值、P95、P99，而不只看吞吐。
5. 使用独立压测机，区分客户端与服务器资源竞争。
6. 增加连接中途断开、慢请求、队列满、超时与 fd 高频复用测试。
7. 使用 ThreadSanitizer 验证共享状态，使用 AddressSanitizer 验证定时器和连接生命周期。

## 14. 最终记忆锚点

只记住下面五句话，也能把这个改动讲清楚：

1. **现象**：历史 Reactor 只有 108/359 次；恢复 `-O2` 后，本项目复现 110/119 次，官方复现 149/440 次。
2. **上游证据**：官方从 `7a16b8f9` 引入到固定提交 `4bcf887`，一直使用普通整型标志跨线程同步；官方源码已实际构建压测。
3. **根因**：数据竞争使 GCC `-O2` 删除 `improv` 等待条件并破坏活性；即使 atomic 恢复正确等待，事件线程仍被串行化并消耗高 CPU。
4. **方案**：不可变 `work_item` + 成功路径直接重置 `EPOLLONESHOT` + 失败路径 `eventfd` 回主循环 + generation 过滤迟到关闭。
5. **结果**：正式修复达到约 38 万至 39 万次/30 秒，相对故障版恢复约 3217 至 3583 倍；相对 atomic 最小修复，CPU 再降低约 35% 至 39%，两组均 0 失败。
