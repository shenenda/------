# http_conn.h / http_conn.cpp 核心知识点与面试拷问

> 学习范围：`http/http_conn.h`、`http/http_conn.cpp`，并结合 `webserver.cpp::dealwithread/dealwithwrite`、`threadpool/threadpool.h::run` 理解真实调用链。
>
> 核心定位：`http_conn` 不是整个服务器，而是“单个客户端 HTTP 连接的处理对象”。一个 `http_conn` 对象负责一个客户端连接上的读缓冲区、HTTP 解析状态、响应生成、文件映射和发送进度。

## 1. 先把角色说清楚

`WebServer` 是总调度器，负责监听、accept、epoll 分发。

`http_conn` 是单连接处理器，负责一个客户端连接内部的状态：

```text
客户端 socket fd
    -> http_conn 对象
    -> m_read_buf 保存请求报文
    -> 状态机解析 HTTP
    -> do_request 定位静态文件或处理登录注册
    -> m_write_buf 拼响应头
    -> mmap 映射文件内容
    -> writev 发送响应
```

一句话：

> `http_conn` 把“一个 socket 上收到的 HTTP 请求”转成“一个可发送的 HTTP 响应”。

## 2. 一个容易混淆的关键点

当前项目里，`process()` 本身不是负责 `recv()` 的函数。

真实调用关系要分两种并发模型：

### Proactor 模式，`actor_model == 0`

```text
WebServer::dealwithread()
    -> users[sockfd].read_once()
    -> m_pool->append_p(users + sockfd)
    -> threadpool::run()
    -> request->process()
```

这里是主线程先完成 socket 读取，工作线程只负责 `process()` 解析和生成响应。

### Reactor 模式，`actor_model == 1`

```text
WebServer::dealwithread()
    -> m_pool->append(users + sockfd, 0)
    -> threadpool::run()
    -> request->read_once()
    -> connectionRAII
    -> request->process()
```

这里是工作线程自己读取 socket，然后再调用 `process()`。

所以应该这样理解：

| 函数 | 真正职责 |
| --- | --- |
| `read_once()` | 从 socket 读取客户端报文，写入 `m_read_buf` |
| `process()` | 解析已经读到缓冲区里的 HTTP 报文，并生成响应 |
| `write()` | 把响应数据发回客户端 |

你现在觉得流程乱，很大一部分原因就是这三个函数容易被混成一个函数。

## 3. `http_conn.h` 是这两个文件的地图

头文件里最重要的是三类东西。

### 3.1 状态枚举

```cpp
enum METHOD
enum CHECK_STATE
enum HTTP_CODE
enum LINE_STATUS
```

它们分别回答四个问题：

| 枚举 | 回答的问题 |
| --- | --- |
| `METHOD` | HTTP 方法是什么？GET 还是 POST？ |
| `CHECK_STATE` | 主状态机正在解析请求行、请求头，还是请求体？ |
| `HTTP_CODE` | 解析/业务处理的结果是什么？请求未完成、请求成功、文件不存在、权限不足、内部错误？ |
| `LINE_STATUS` | 当前行有没有读完整？格式是否正确？ |

### 3.2 对外接口

这些函数是 `WebServer` 或线程池会调用的：

| 函数 | 作用 |
| --- | --- |
| `init(sockfd, addr, ...)` | 新连接建立后初始化该连接对象 |
| `close_conn()` | 从 epoll 移除 fd 并关闭 socket |
| `read_once()` | 从 socket 读取数据到读缓冲区 |
| `process()` | 解析请求并生成响应 |
| `write()` | 发送响应数据 |
| `initmysql_result()` | 服务器启动时预加载用户表 |
| `get_address()` | 返回客户端地址，主要用于日志 |

### 3.3 内部工具函数

这些是 `http_conn` 自己内部拆出来的步骤：

| 函数 | 作用 |
| --- | --- |
| `process_read()` | HTTP 请求解析总入口 |
| `parse_line()` | 从缓冲区里切出一行 |
| `parse_request_line()` | 解析请求行 |
| `parse_headers()` | 解析请求头 |
| `parse_content()` | 解析 POST 请求体 |
| `do_request()` | 根据 URL 做业务处理，定位文件或登录注册 |
| `process_write()` | 根据处理结果生成响应 |
| `add_response()` | 往写缓冲区格式化追加内容 |
| `unmap()` | 释放 `mmap` 映射 |
| `parse_form_data()` | 从表单字符串里提取用户名和密码 |

## 4. 成员变量如何理解

`http_conn` 的成员变量可以按“请求、解析、响应、连接、数据库”五类理解。

### 4.1 连接相关

| 成员 | 含义 |
| --- | --- |
| `m_sockfd` | 当前客户端连接的 socket fd |
| `m_address` | 客户端地址 |
| `m_epollfd` | 全服务器共享的 epoll fd，静态成员 |
| `m_user_count` | 当前活跃连接数，静态成员 |
| `m_TRIGMode` | 当前连接的 LT/ET 触发模式 |

为什么 `m_epollfd` 是静态成员？

因为所有 `http_conn` 对象共享同一个 epoll 实例，而不是每个连接自己创建一个 epoll。

### 4.2 读缓冲区和解析游标

| 成员 | 含义 |
| --- | --- |
| `m_read_buf` | 保存客户端发来的 HTTP 请求报文 |
| `m_read_idx` | 已经读取到缓冲区里的字节数 |
| `m_checked_idx` | 解析器已经扫描到的位置 |
| `m_start_line` | 当前行的起始位置 |

理解这三个下标非常重要：

```text
m_read_buf
|---------------- 已经读到的数据 ----------------|
0              m_start_line    m_checked_idx    m_read_idx
```

- `m_read_idx`：网络读到哪里。
- `m_checked_idx`：解析检查到哪里。
- `m_start_line`：当前准备解析的这一行从哪里开始。

### 4.3 HTTP 字段

| 成员 | 含义 |
| --- | --- |
| `m_method` | GET / POST |
| `m_url` | 请求路径 |
| `m_version` | HTTP 版本 |
| `m_host` | Host 头 |
| `m_content_length` | 请求体长度 |
| `m_linger` | 是否长连接 |
| `cgi` | 是否需要处理登录/注册 |
| `m_string` | POST 表单请求体 |

### 4.4 响应相关

| 成员 | 含义 |
| --- | --- |
| `m_write_buf` | 响应头和错误响应正文缓冲区 |
| `m_write_idx` | 写缓冲区已经写了多少 |
| `m_real_file` | 本地真实文件路径 |
| `m_file_address` | `mmap` 映射后的文件内存地址 |
| `m_file_stat` | 文件状态信息 |
| `m_iv[2]` | `writev` 使用的分散写数组 |
| `bytes_to_send` | 还需要发送多少字节 |
| `bytes_have_send` | 已经发送多少字节 |

### 4.5 数据库相关

| 成员 | 含义 |
| --- | --- |
| `mysql` | 当前请求临时借用的 MySQL 连接 |
| `sql_user/sql_passwd/sql_name` | 数据库连接配置 |
| 全局 `users` map | 预加载的用户名密码表 |
| 全局 `m_lock` | 注册新用户时保护 map 和数据库写入 |

## 5. 新连接初始化

`http_conn::init(int sockfd, ...)` 是新 TCP 连接建立后调用的外部初始化：

```cpp
m_sockfd = sockfd;
m_address = addr;
addfd(m_epollfd, sockfd, true, m_TRIGMode);
m_user_count++;
doc_root = root;
m_TRIGMode = TRIGMode;
m_close_log = close_log;
init();
```

它做三件事：

1. 保存连接 fd 和客户端地址。
2. 把连接 fd 注册进 epoll，并开启 `EPOLLONESHOT`。
3. 调用内部无参 `init()` 重置读写缓冲区和解析状态。

注意一个当前代码层面的隐患：

```cpp
addfd(m_epollfd, sockfd, true, m_TRIGMode);
m_TRIGMode = TRIGMode;
```

这里先使用 `m_TRIGMode`，后给它赋值。更合理的顺序应该是先 `m_TRIGMode = TRIGMode`，再 `addfd(...)`。否则新连接注册 epoll 时可能使用的是旧值或未初始化值。

这属于小范围代码问题，但这份笔记先从学习角度提醒你：看代码时不要只顺着注释读，还要看成员变量什么时候被赋值、什么时候被使用。

## 6. 读数据：`read_once()`

`read_once()` 把 socket 中的数据读入 `m_read_buf`。

### LT 模式

```cpp
bytes_read = recv(...);
m_read_idx += bytes_read;
```

LT 模式下只读一次。因为只要 socket 里还有数据，epoll 后面还会继续提醒。

### ET 模式

```cpp
while (true) {
    bytes_read = recv(...);
    if (errno == EAGAIN || errno == EWOULDBLOCK) break;
}
```

ET 模式下必须循环读，直到 `EAGAIN/EWOULDBLOCK`。

原因：

> ET 只通知一次，如果这次没有把内核缓冲区读空，剩下的数据可能再也不会触发读事件。

面试常问：

> 为什么 ET 模式必须配合非阻塞 socket？

答：因为 ET 模式要求循环读到没有数据。如果 socket 是阻塞的，读到最后没有数据时线程会卡死。

## 7. 从状态机：`parse_line()`

HTTP 报文是文本协议，请求头每一行以 `\r\n` 结束。

`parse_line()` 的作用是：

> 从连续的 `m_read_buf` 中找出一行完整文本，把行尾的 `\r\n` 改成 `\0\0`，让这一行变成 C 字符串，供后续解析。

返回值：

| 返回值 | 含义 |
| --- | --- |
| `LINE_OK` | 成功读到完整一行 |
| `LINE_BAD` | 行格式错误 |
| `LINE_OPEN` | 当前行还没读完整，需要继续等数据 |

为什么要把 `\r\n` 改成 `\0\0`？

因为后面的 `parse_request_line()`、`parse_headers()` 都是按 C 字符串处理的。把行尾改成字符串结束符后，每一行就可以独立解析。

## 8. 主状态机：`process_read()`

`process_read()` 是 HTTP 请求解析总入口，它根据 `m_check_state` 分阶段解析：

```text
CHECK_STATE_REQUESTLINE
    -> parse_request_line()
    -> 解析方法、URL、HTTP版本
    -> 成功后切到 CHECK_STATE_HEADER

CHECK_STATE_HEADER
    -> parse_headers()
    -> 解析 Connection、Content-Length、Host
    -> 空行表示头部结束
    -> 没有 body：GET_REQUEST
    -> 有 body：切到 CHECK_STATE_CONTENT

CHECK_STATE_CONTENT
    -> parse_content()
    -> 等 body 长度达到 Content-Length
    -> GET_REQUEST
```

当完整请求解析完成后，就调用：

```cpp
return do_request();
```

这就是典型的“主状态机 + 从状态机”：

| 状态机 | 函数 | 作用 |
| --- | --- | --- |
| 从状态机 | `parse_line()` | 判断一行是否完整 |
| 主状态机 | `process_read()` | 判断当前该解析请求行、请求头还是请求体 |

## 9. 请求行解析

请求行例子：

```http
GET /index.html HTTP/1.1
```

`parse_request_line()` 要解析出三部分：

| 字段 | 存到哪里 |
| --- | --- |
| 方法 `GET/POST` | `m_method` |
| URL `/index.html` | `m_url` |
| 版本 `HTTP/1.1` | `m_version` |

核心技巧：

- 用 `strpbrk` 找空格或制表符。
- 用 `*m_url++ = '\0'` 把字符串切断。
- 用 `strcasecmp` 忽略大小写比较。
- 如果 URL 是 `/`，默认拼成 `/home.html`。

你需要掌握的是思想，不是背 `strpbrk` 的每个细节：

> HTTP 请求行本质就是一行字符串，通过空格拆成 method、url、version 三段。

## 10. 请求头解析

`parse_headers()` 主要关心三类头：

| 请求头 | 作用 |
| --- | --- |
| `Connection:` | 判断是否 `keep-alive` 长连接 |
| `Content-Length:` | 判断 POST 请求体长度 |
| `Host:` | HTTP/1.1 必需字段 |

当读到空行时：

```cpp
if (text[0] == '\0')
```

说明请求头结束。

如果 `m_content_length != 0`，说明后面还有请求体，要切换到 `CHECK_STATE_CONTENT`。

如果没有请求体，说明 GET 请求已经完整，可以返回 `GET_REQUEST`，进入 `do_request()`。

## 11. 请求体解析

`parse_content()` 处理 POST 表单。

核心判断：

```cpp
if (m_read_idx >= (m_content_length + m_checked_idx))
```

意思是：已经读到的数据长度是否覆盖了整个请求体。

如果够了：

```cpp
text[m_content_length] = '\0';
m_string = text;
return GET_REQUEST;
```

`m_string` 指向请求体内容，例如：

```text
user=tom&password=123456
```

后面 `do_request()` 会调用 `parse_form_data()` 提取用户名和密码。

## 12. 业务处理：`do_request()`

`do_request()` 做两类事情：

1. 静态资源请求：根据 URL 拼接本地文件路径，检查文件，`mmap` 映射。
2. 登录/注册请求：解析表单，查数据库或插入数据库，然后跳转到对应页面。

### 静态资源流程

```text
doc_root + m_url
    -> m_real_file
    -> stat 检查文件
    -> open 打开文件
    -> mmap 映射文件
    -> close fd
    -> return FILE_REQUEST
```

`stat()` 判断：

| 情况 | 返回 |
| --- | --- |
| 文件不存在 | `NO_RESOURCE` |
| 没有读权限 | `FORBIDDEN_REQUEST` |
| 是目录 | `BAD_REQUEST` |
| 正常文件 | `FILE_REQUEST` |

### 登录/注册流程

项目用 URL 标记区分页面：

| URL 标记 | 含义 |
| --- | --- |
| `/0` | 注册页面 |
| `/1` | 登录页面 |
| `/2` | 登录校验 |
| `/3` | 注册校验 |
| `/5` | 图片页面 |
| `/6` | 视频页面 |
| `/7` | 粉丝页面 |

登录校验：

```cpp
if (users.find(name) != users.end() && users[name] == password)
    strcpy(m_url, "/welcome.html");
else
    strcpy(m_url, "/logError.html");
```

注册校验：

```cpp
if (users.find(name) == users.end()) {
    mysql_query(mysql, sql_insert);
    users.insert(...);
    strcpy(m_url, "/log.html");
} else {
    strcpy(m_url, "/registerError.html");
}
```

这里体现了两个点：

- 预加载用户表到内存 map，提高登录校验速度。
- 注册时仍需要写数据库，并更新内存 map。

## 13. `mmap` 和 `writev`

这是 `http_conn` 中非常有价值的性能点。

普通发送静态文件可能是：

```text
read 文件到用户态缓冲区
    -> 拼响应头
    -> write 响应头
    -> write 文件内容
```

项目使用：

```text
mmap 文件
    -> m_iv[0] 指向响应头 m_write_buf
    -> m_iv[1] 指向文件映射地址 m_file_address
    -> writev 一次发送两块内存
```

好处：

1. `mmap` 避免手动把文件内容读入用户缓冲区。
2. `writev` 支持分散写，一次系统调用发送响应头和文件内容。
3. 不受 `WRITE_BUFFER_SIZE` 太小的限制，大文件不需要塞进 `m_write_buf`。

面试可以这样讲：

> 响应头是动态生成的，放在 `m_write_buf`；文件内容通过 `mmap` 映射到内存。两块内存不连续，所以用 `writev` 一次性发送，减少系统调用次数。

## 14. 响应生成：`process_write()`

`process_write()` 根据 `HTTP_CODE` 生成不同响应。

| `HTTP_CODE` | 响应 |
| --- | --- |
| `INTERNAL_ERROR` | 500 |
| `BAD_REQUEST` | 当前代码里走 404 响应，语义上更应该是 400 |
| `FORBIDDEN_REQUEST` | 403 |
| `FILE_REQUEST` | 200，发送静态文件 |

成功文件请求时：

```cpp
m_iv[0].iov_base = m_write_buf;
m_iv[0].iov_len = m_write_idx;
m_iv[1].iov_base = m_file_address;
m_iv[1].iov_len = m_file_stat.st_size;
m_iv_count = 2;
bytes_to_send = m_write_idx + m_file_stat.st_size;
```

错误响应和空文件只使用一块缓冲区：

```cpp
m_iv[0].iov_base = m_write_buf;
m_iv[0].iov_len = m_write_idx;
m_iv_count = 1;
bytes_to_send = m_write_idx;
```

## 15. 响应发送：`write()`

`write()` 使用 `writev()` 循环发送。

关键变量：

| 变量 | 含义 |
| --- | --- |
| `bytes_to_send` | 剩余待发送字节 |
| `bytes_have_send` | 已经发送字节 |
| `m_iv[0]` | 响应头 |
| `m_iv[1]` | 文件内容 |

如果内核发送缓冲区满：

```cpp
if (errno == EAGAIN) {
    modfd(m_epollfd, m_sockfd, EPOLLOUT, m_TRIGMode);
    return true;
}
```

意思是：现在发不完，先返回，等下次 socket 可写时继续发。

发送完后：

```cpp
unmap();
modfd(m_epollfd, m_sockfd, EPOLLIN, m_TRIGMode);
if (m_linger) {
    init();
    return true;
} else {
    return false;
}
```

- 长连接：重置状态，继续等下一个请求。
- 短连接：返回 false，让上层关闭连接。

## 16. `process()` 到底做什么

`process()` 是 `http_conn` 的业务入口：

```cpp
HTTP_CODE read_ret = process_read();

if (read_ret == NO_REQUEST) {
    modfd(..., EPOLLIN, ...);
    return;
}

bool write_ret = process_write(read_ret);

if (!write_ret) {
    close_conn();
}

modfd(..., EPOLLOUT, ...);
```

它做的是：

1. 调 `process_read()` 解析请求。
2. 如果请求还不完整，继续监听读事件。
3. 如果请求完整，根据结果生成响应。
4. 把 fd 改成监听写事件，等 epoll 通知可写后发送响应。

再次强调：

> 在当前项目实现里，`process()` 不直接 `recv()`，它解析的是已经读入 `m_read_buf` 的数据。

## 17. 这两个文件体现的 C++ / 系统编程能力

`http_conn` 不主要体现高级模板或现代 C++，它体现的是系统级 C++ 能力：

| 能力 | 代码体现 |
| --- | --- |
| C++ 封装 | 用 `http_conn` 类封装单连接状态 |
| 状态机设计 | `CHECK_STATE` + `LINE_STATUS` 解析 HTTP |
| 指针和缓冲区管理 | `char*`、`m_read_buf`、`m_write_buf`、游标 |
| Linux 网络编程 | `recv`、`epoll_ctl`、非阻塞 fd |
| 高性能 IO | `mmap`、`writev` |
| 资源管理 | `unmap()`、`close_conn()`、数据库 RAII 借用 |
| 并发意识 | `EPOLLONESHOT`、全局用户表锁 |
| 协议理解 | HTTP 请求行、请求头、请求体、响应状态码 |
| 数据库交互 | `mysql_query`、`mysql_real_escape_string` |

## 18. 当前代码值得注意的隐患

这些点面试时不一定主动展开，但如果被问“项目有什么不足”，你可以说：

1. `http_conn::init()` 中 `addfd()` 使用 `m_TRIGMode` 的时机早于赋值，触发模式可能不符合配置。
2. `process_write(BAD_REQUEST)` 当前返回的是 404 标题和正文，语义上更应该返回 400。
3. `process()` 的注释如果写成“读取客户端数据并解析”容易误导，准确说法是“解析已读缓冲区并生成响应”。
4. 全局 `users` map 用于用户名密码缓存，但 `http_conn` 里还有一个 `m_users` 成员没有实际使用，结构上有冗余。
5. 注册时对全局 map 加锁，但登录时读取 map 没加锁，严格并发语义下仍有数据竞争风险。
6. 使用大量 C 风格字符串和固定大小缓冲区，对越界和字符串截断要非常敏感。
7. `mmap()`、`open()` 等系统调用的失败分支不够完整，工程健壮性还有提升空间。
8. 登录注册的 URL 标记 `/2`、`/3` 属于教学项目写法，真实项目应该使用更清晰的路由。

## 19. 面试拷问点

### Q1：`http_conn` 在整个项目里负责什么？

答：它负责单个客户端连接的 HTTP 请求读取状态、解析状态、响应生成和发送进度。`WebServer` 负责事件分发，`http_conn` 负责连接内部的协议处理。

### Q2：`process()` 和 `read_once()` 的区别是什么？

答：`read_once()` 从 socket 读取数据到 `m_read_buf`；`process()` 解析 `m_read_buf` 中已有的数据并生成响应。Proactor 模式下主线程先调用 `read_once()`，工作线程调用 `process()`；Reactor 模式下工作线程先 `read_once()` 再 `process()`。

### Q3：HTTP 解析为什么要用状态机？

答：TCP 是字节流协议，一次 `recv()` 不一定读到完整 HTTP 请求。状态机可以记住已经解析到请求行、请求头还是请求体，下次数据到来时接着解析。

### Q4：主状态机和从状态机分别是什么？

答：从状态机是 `parse_line()`，负责判断是否读到完整一行；主状态机是 `process_read()`，负责根据当前阶段解析请求行、请求头或请求体。

### Q5：为什么 ET 模式下 `read_once()` 要循环读？

答：ET 模式只在状态变化时通知一次，必须一次性把 socket 缓冲区读空，直到 `EAGAIN/EWOULDBLOCK`，否则剩余数据可能不会再触发事件。

### Q6：`EPOLLONESHOT` 有什么作用？

答：保证一个连接同一时刻只被一个线程处理。事件触发一次后 fd 会被暂时禁用，处理完成后必须 `modfd()` 重新注册。

### Q7：为什么使用 `mmap`？

答：`mmap` 把文件映射到进程地址空间，避免手动 `read` 文件到用户缓冲区，适合静态文件服务器发送文件内容。

### Q8：为什么使用 `writev`？

答：响应头和文件内容在两块不连续内存中，`writev` 可以一次系统调用发送多块内存，减少系统调用次数。

### Q9：长连接如何实现？

答：解析请求头里的 `Connection: keep-alive`，设置 `m_linger = true`。响应发送完后不关闭 socket，而是调用 `init()` 重置连接状态，再重新监听 `EPOLLIN`。

### Q10：数据库登录注册怎么做？

答：服务器启动时把用户表加载到全局 map。登录时查 map 判断用户名密码。注册时先查 map，不存在则执行 SQL 插入数据库，并同步更新 map。

### Q11：为什么 `mysql` 是裸指针？

答：它只是临时借用连接池中的 `MYSQL*`，生命周期由连接池管理。线程处理请求时通过 `connectionRAII` 借出连接，作用域结束自动归还。

### Q12：这个模块有哪些不足？

答：可以从触发模式赋值顺序、状态码语义、全局 map 并发读写、C 字符串缓冲区风险、系统调用错误处理、教学式 URL 路由等方面回答。

## 20. 复习顺序

建议按这个顺序重新看：

1. 先看 `http_conn.h`：只看成员变量，搞清楚一个连接要保存哪些状态。
2. 看 `read_once()`：搞清 LT/ET 读数据差异。
3. 看 `parse_line()`：理解从状态机如何切行。
4. 看 `process_read()`：理解主状态机如何驱动请求行、请求头、请求体。
5. 看 `do_request()`：理解静态文件和登录注册业务。
6. 看 `process_write()`：理解响应如何组装。
7. 看 `write()`：理解 `writev` 如何处理未发送完的数据。
8. 回到 `webserver.cpp::dealwithread/dealwithwrite`：把 `http_conn` 放回项目主流程。

