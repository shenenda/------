# 2026-07-16 TinyWebServer `-O0` 受控复现实验

> **结论状态：已被后续 `-O2` 取证实验修正。** 本目录的原始测试数据仍然有效，但“108/359 无法复现”的原因是本轮构建没有使用历史命令中的 `-O2`，不能再作为最终根因结论。最终报告见 [`../2026-07-16-o2-race-forensics/README.md`](../2026-07-16-o2-race-forensics/README.md)。

## 结论

本次实验补做了官方 TinyWebServer 的独立构建和四组压测，并重新构建了本项目修复前、修复后的 Reactor 版本。它使用 `-g` 或未指定优化级别的构建，实际属于 `-O0` 对照。

本目录能够证明：

1. 官方源码存在主事件线程等待和普通整数跨线程同步的问题。
2. 在 `-O0` 下，官方原版与修复前提交都能达到约 1.24 万到 1.32 万 req/s；这是编译器反复读取 `improv` 后的偶然可运行表现，不能证明代码符合 C++ 内存模型。
3. 在同样 `-O0` 构建下，正式修复版把 Reactor 的服务器 CPU 降低约 34% 到 37%，ET/ET 吞吐基本持平，LT/LT 吞吐在本轮下降约 6.4%。
4. 本实验不能否定历史 108/359 结果，因为历史实际编译命令带有 `-O2`。后续已在历史提交和官方提交上用 `-O2` 重新复现百次级低吞吐，并用反汇编确认等待条件被优化删除。

## 实验环境

- 日期：2026-07-16
- 主机：阿里云服务器 `didida`
- 内核：Linux `5.15.0-142-generic`
- 逻辑 CPU：2
- MySQL：运行中
- 编译：`g++`，`-g -lpthread -lmysqlclient`
- 压测端与服务端：同一台主机，通过 `127.0.0.1`
- Webbench：100 客户端，30 秒
- 服务端线程：默认 8
- 日志：`-c 1` 关闭
- 端口：9016

统一压测命令：

```bash
webbench -c 100 -t 30 http://127.0.0.1:9016/
```

## 官方原版四组结果

官方仓库：<https://github.com/qinguoyi/TinyWebServer>

固定提交：`4bcf88762f85135e9bd46a1032e815e458de6f2e`

官方临时副本只替换了 `main.cpp` 中用户名、密码和数据库名三行，以连接现有 MySQL。`git diff --numstat` 为：

```text
3  3  main.cpp
```

并发实现、线程池、epoll、定时器、页面文件和编译参数均保持官方版本。

| 官方模式 | 成功请求/30s | 吞吐 | 服务器 CPU | 失败 | 压测后 HTTP |
|---|---:|---:|---:|---:|---:|
| Reactor LT + LT | 397046 | 13234.87 req/s | 114.31% | 0 | 200 |
| Reactor ET + ET | 371031 | 12367.70 req/s | 116.87% | 0 | 200 |
| Proactor LT + LT | 571452 | 19048.40 req/s | 78.23% | 0 | 200 |
| Proactor ET + ET | 510682 | 17022.73 req/s | 68.77% | 0 | 200 |

在这一次运行中：

- 官方 Proactor LT/LT 相比 Reactor LT/LT，吞吐高 43.93%，服务器 CPU 低 31.56%。
- 官方 Proactor ET/ET 相比 Reactor ET/ET，吞吐高 37.64%，服务器 CPU 低 41.16%。

这个比较说明官方 Reactor 路径在该场景下 CPU 效率较低，但 Reactor 与 Proactor 的 IO 分工不同，因此不能把全部差距单独归因于 `while (true)`。

## 对旧结果判断的修正

仓库原有文件记录的是：

- LT/LT：359 成功/30 秒；
- ET/ET：108 成功/30 秒。

本目录使用 `-O0` 后，官方原版和 `a57a275` 都达到每 30 秒约 37 万到 40 万成功请求。当时据此把旧结果判断为“不可复现”，这个判断不完整。

后续从原始会话恢复出历史编译命令，其中明确包含 `-O2`。按该命令重建后：

- `a57a275 -O2`：LT/LT 110、ET/ET 119 成功请求/30 秒；
- 官方 `4bcf887 -O2`：LT/LT 355、ET/ET 228 成功请求/30 秒；
- 两组均为近零 CPU、零失败，并在压测中很快失去可推进工作。

所以旧文件不是凭空出现的异常。准确结论是：普通 `int` 跨线程数据竞争在 `-O0` 下被掩盖，在 GCC 11.4 的 `-O2` 下使 `improv` 等待判断被删除，导致 Reactor 活性崩溃。旧 ELF 没有被 Git 跟踪，无法逐字节还原，但故障机制和数量级已经重复复现。

## 文件说明

每个测试包含：

- `*-metadata.txt`：提交、命令、页面 smoke、服务器 CPU 和 Webbench 退出码；
- `*-webbench.txt`：Webbench 原始输出；
- `*-cpu.txt`：`pidstat` 原始采样；
- `run_benchmark.sh`：本次使用的统一测试脚本。

Webbench 1.5 的 `bench()` 最后返回局部变量 `i`，不是固定返回 0，因此即使生成完整结果也可能出现非零退出码。脚本同时检查 `Speed` 和 `Requests` 两行来判断结果是否完整，并保留原退出码供审计。

## 结果边界

- 压测端与服务端在同一台 2 核机器，会竞争 CPU。
- 每个模式只保留一轮正式结果，没有均值、方差和置信区间。
- Webbench 不提供 P95/P99 延迟。
- 官方原版与本项目首页大小不同，不能跨页面直接评价代码改动。
- 本实验能证明 `-O0` 下等待循环的 CPU 成本和正式修复后的 CPU 降幅；关于历史低吞吐的最终解释必须结合后续 `-O2` 取证报告。
