# 2026-07-16 TinyWebServer 受控复现实验

## 结论

本次实验补做了官方 TinyWebServer 的独立构建和四组压测，并重新构建了本项目修复前、修复后的 Reactor 版本。

结论与此前仅依据旧结果文件得到的结论不同：

1. 官方源码确实存在主事件线程自旋和普通整数跨线程同步的问题。
2. 官方原版在本机仍能达到约 1.24 万到 1.32 万 req/s，没有复现 30 秒只完成 108 或 359 个请求。
3. 修复前提交 `a57a275` 从源码重新编译后同样达到约 1.25 万到 1.32 万 req/s。
4. 因此，旧的 `reactor-ltlt.txt` 和 `reactor-etet.txt` 是不可复现的历史异常，不能作为“修复提升 1000 到 3500 倍”的因果证据。
5. 在同一项目页面、同一构建方法下，当前修复版把 Reactor 的服务器 CPU 降低约 34% 到 37%；ET/ET 吞吐基本持平，LT/LT 吞吐下降约 6.4%。主要收益是同步正确性、事件循环不再忙等和 CPU 效率，而不是三个数量级的吞吐提升。

## 实验环境

- 日期：2026-07-16
- 主机：阿里云服务器 `didida`
- 内核：Linux `5.15.0-142-generic`
- 逻辑 CPU：2
- MySQL：运行中
- 编译：`g++`，`-g -lpthread -lmysqlclient`
- 压测端与服务端：同一台主机，通过 `127.0.0.1`
- Webbench：100 客户端，30 秒
- 服务端线程：默认 8
- 日志：`-c 1` 关闭
- 端口：9016

统一压测命令：

```bash
webbench -c 100 -t 30 http://127.0.0.1:9016/
```

## 官方原版四组结果

官方仓库：<https://github.com/qinguoyi/TinyWebServer>

固定提交：`4bcf88762f85135e9bd46a1032e815e458de6f2e`

官方临时副本只替换了 `main.cpp` 中用户名、密码和数据库名三行，以连接现有 MySQL。`git diff --numstat` 为：

```text
3  3  main.cpp
```

并发实现、线程池、epoll、定时器、页面文件和编译参数均保持官方版本。

| 官方模式 | 成功请求/30s | 吞吐 | 服务器 CPU | 失败 | 压测后 HTTP |
|---|---:|---:|---:|---:|---:|
| Reactor LT + LT | 397046 | 13234.87 req/s | 114.31% | 0 | 200 |
| Reactor ET + ET | 371031 | 12367.70 req/s | 116.87% | 0 | 200 |
| Proactor LT + LT | 571452 | 19048.40 req/s | 78.23% | 0 | 200 |
| Proactor ET + ET | 510682 | 17022.73 req/s | 68.77% | 0 | 200 |

在这一次运行中：

- 官方 Proactor LT/LT 相比 Reactor LT/LT，吞吐高 43.93%，服务器 CPU 低 31.56%。
- 官方 Proactor ET/ET 相比 Reactor ET/ET，吞吐高 37.64%，服务器 CPU 低 41.16%。

这个比较说明官方 Reactor 路径在该场景下 CPU 效率较低，但 Reactor 与 Proactor 的 IO 分工不同，因此不能把全部差距单独归因于 `while (true)`。

## 修复前后同项目对照

为了排除官方页面只有 586 字节、而本项目页面为 6573 字节造成的干扰，修复前后对照只在本项目两个提交之间进行。

- 修复前：`a57a275b4866cbc309f51f00da24dcc081030775`
- 修复后：`31bdceec816aba9c77a8a780dce38642d566e9ec`，其服务器源码等同于修复提交 `7ab4ec7`
- 两个提交都从源码重新执行相同的 `g++` 命令构建
- 两组首页均为 6573 字节

| 模式 | 版本 | 成功请求/30s | 吞吐 | 服务器 CPU | 失败 |
|---|---|---:|---:|---:|---:|
| LT + LT | 修复前重编译 | 394870 | 13162.33 req/s | 115.08% | 0 |
| LT + LT | 当前修复版 | 369548 | 12318.27 req/s | 76.20% | 0 |
| ET + ET | 修复前重编译 | 373618 | 12453.93 req/s | 116.96% | 0 |
| ET + ET | 当前修复版 | 374153 | 12471.77 req/s | 74.20% | 0 |

修复前后变化：

| 模式 | 吞吐变化 | CPU 变化 | 每 CPU 百分点吞吐变化 |
|---|---:|---:|---:|
| LT + LT | -6.41% | -33.79% | +41.34% |
| ET + ET | +0.14% | -36.56% | +57.85% |

这是一轮测试，不应把小幅吞吐差异解释成稳定性能结论。但 CPU 降幅与自旋移除的设计目标一致，且两组均保持 0 失败、压测后 HTTP 200。

## 为什么旧结果不能继续使用

仓库原有文件记录的是：

- LT/LT：359 成功/30 秒；
- ET/ET：108 成功/30 秒。

本次使用同一台服务器重新执行后：

- 官方原版没有复现；
- `a57a275` 从源码重编译后也没有复现；
- 两者都稳定在每 30 秒约 37 万到 40 万成功请求。

目前没有足够证据精确还原旧异常的唯一原因。可能因素包括旧二进制未对应当时源码、服务器进程状态异常或测试流程不一致。正确做法是把它标记为不可复现的历史异常，而不是继续把它包装成修复效果。

## 文件说明

每个测试包含：

- `*-metadata.txt`：提交、命令、页面 smoke、服务器 CPU 和 Webbench 退出码；
- `*-webbench.txt`：Webbench 原始输出；
- `*-cpu.txt`：`pidstat` 原始采样；
- `run_benchmark.sh`：本次使用的统一测试脚本。

Webbench 1.5 的 `bench()` 最后返回局部变量 `i`，不是固定返回 0，因此即使生成完整结果也可能出现非零退出码。脚本同时检查 `Speed` 和 `Requests` 两行来判断结果是否完整，并保留原退出码供审计。

## 结果边界

- 压测端与服务端在同一台 2 核机器，会竞争 CPU。
- 每个模式只保留一轮正式结果，没有均值、方差和置信区间。
- Webbench 不提供 P95/P99 延迟。
- 官方原版与本项目首页大小不同，不能跨页面直接评价代码改动。
- 本实验能证明自旋代码存在、修复后 CPU 成本下降；不能证明所有硬件和流量下吞吐都会提高。
